# 🤝 貢献ガイド（CONTRIBUTING.md）

このドキュメントは、プロジェクトへの貢献方法や開発フローを説明するためのものです。
以下のルールと手順に従って、スムーズで信頼性の高い開発サイクルを一緒に築いていきましょう。

---

## 🐞 Issue の作成ルール

Issueとは、プロジェクトにおけるタスクやバグ、機能追加の提案などを管理する投稿のことです。

### ✅ 使用テンプレート

Issueは以下のテンプレートから作成してください：

- `🐞 Bug_Report`：バグ報告
- `✨ Feature_Request`：機能追加提案
- `🔧 Improvement`：既存機能の改善

テンプレート選択は、**「テンプレートから選択」** プルダウンから可能です。

---

## 🚀 Merge Request（MR）ルール

Merge Request（MR）は、コードの変更を `main` ブランチに統合するためのリクエストです。
本プロジェクトは [GitHub Flow](https://docs.github.com/ja/get-started/quickstart/github-flow) をベースにしています。

### ✅ 基本フロー

1. `main` ブランチから `feature/xxx` や `fix/xxx` ブランチを切る
2. 作業をコミット & Push
3. **MR = リリース単位** として、テンプレートを使って MR を作成
4. CI を通過し、レビューを経て `main` にマージ
5. 必要に応じて Git タグを作成（例: `v1.3.0`）

---

### 📝 MR テンプレートについて

MRには自動で `Default.md` テンプレートが読み込まれます。  
以下の内容を必ず記載してください：

- 🎯 このMRの目的
- 📝 `CHANGELOG.md` として使える Markdown 形式の変更履歴
- 🔖 リリース予定（ブランチ、バージョン、日付）
- ✅ セルフチェックリスト（README更新、Unityバージョン、フォーマッタなど）

**テンプレート例： `.gitlab/merge_request_templates/Default.md` を参照**

---

### ✅ セルフチェック項目（抜粋）

- `.editorconfig` および StyleCop に準拠しているか？
- `package.json` の `version` を更新したか？
- `CHANGELOG.md` に変更履歴を記載したか？
- Unity Editorバージョンがチームルールに合っているか？

**すべての項目はMRテンプレ内でチェックボックス形式で確認できます。**

---

## 🧪 テストとCIについて

- すべての変更は CI を通過していることが前提です。
- `CHANGELOG.md` や `version` の更新漏れがある場合、CIエラーになります。
- CIで自動タグ、CHANGELOG生成などを行う予定です。

---

## 📦 リリース・タグ運用

- 本リポジトリでは、**mainへのマージ = リリース** です。
- マージ後に `vX.X.X` 形式でタグを打ち、必要に応じてパッケージ公開（npm / UPM）を行います。
- Git タグの作成は、`CHANGELOG.md` に準じて実施してください。

---

## 🙋‍♂️ わからないときは？

- 疑問点があれば チームリーダー、テックリーダー (技術・品質担当) へ気軽にメンションをお願いします。
- **MRレビューでのコメント・相互レビュー歓迎**。積極的な意見交換でプロジェクトの品質を一緒に高めましょう！
